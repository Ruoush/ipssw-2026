Glossary

Plain-language definitions of AI and technical terms used throughout the course. Consult this anytime without losing your place.

Agentic AI
AI tools capable of taking multi-step actions on your behalf — not just answering questions, but generating files, writing code, and iterating based on feedback. Claude, ChatGPT, and Gemini are examples used in this course.
API (Application Programming Interface)
A standardized way for software to communicate with other software. When an HTML tool plays a voice prompt, it's using the browser's Web Speech API. You don't need to use APIs directly — the AI handles that in the generated code.
API key
A unique credential — a long string of characters — that proves your identity when connecting to an AI service programmatically. Needed when building applications that call AI directly through code. Not required when using browser-based AI tools like claude.ai or chatgpt.com.
ASI (Artificial Superintelligence)
A hypothetical future form of AI that surpasses human intelligence across all domains. Currently theoretical — no ASI exists. Distinct from today's AI tools, which are highly capable at specific tasks but do not generalize beyond human-level performance.
Back end
The server-side portion of an application — handles data storage, user authentication, and business logic. The tools built in this course have no back end; all logic runs in the browser. If you need to save learner progress across sessions or manage accounts, a back end would be required.
Bash command
An instruction typed into a terminal on Mac or Linux. Bash is the language the terminal understands. Common commands: cd (change directory), ls (list files), git push (upload to GitHub). Not required for this workshop's single-file HTML approach.
Claude
An AI assistant made by Anthropic, designed to be helpful, harmless, and honest. The primary tool recommended in this course for generating HTML simulation tools. Available at claude.ai — free tier available, with more powerful models on Claude Pro ($20/mo).
Claude Code
An agentic AI tool built by Anthropic that runs directly in your terminal or code editor. Unlike the browser interface, Claude Code can read files, run commands, and make changes across your entire project automatically. More powerful for multi-file projects and rapid iteration. Requires Claude Pro.
Codex
An AI model from OpenAI specialized in understanding and generating code. Powers tools like GitHub Copilot. Similar function to Claude Code — provides AI-assisted coding within development environments rather than through a browser chat interface.
Context window
The total amount of text an AI can "hold in mind" during a conversation — your messages, its responses, and any code you paste all count toward this limit. When it fills up, AI performance degrades. Think of it like a whiteboard with limited space.
CSS (Cascading Style Sheets)
The part of an HTML file that controls appearance — colors, fonts, layout, and animations. You won't write CSS directly; the AI generates it as part of your simulation tool.
Front end
The part of a software application that users see and interact with — buttons, layouts, animations, and forms. Everything in your HTML simulation tools is front end. No server required.
GitHub
A web-based platform for storing, sharing, and managing code files. Think of it as Google Drive for code — with complete version history and the ability to share projects publicly or privately. Used in Lesson 7 to host your simulation tools before deploying with Vercel.
Hallucination
When an AI confidently states something incorrect. Common with specific facts, dates, or citations — and a real risk in clinical content. Always verify drug doses, procedural steps, and guidelines in AI-generated tools before using them for training.
HTML (HyperText Markup Language)
The language all websites are built with. An HTML file is a text file your browser knows how to display. No server, no app store, no installation needed — open the file and it works.
JSON (JavaScript Object Notation)
A lightweight text format for storing and exchanging structured data. Looks like nested key-value pairs: {"name": "value"}. Commonly used in configuration files and API responses. You may encounter JSON files in more advanced projects or when connecting AI tools to external services.
Large Reasoning Model (LRM)
A class of AI model trained to think through problems step by step before responding — sometimes called "extended thinking" or chain-of-thought reasoning. Better than standard models at complex logic and multi-step problems. Examples include OpenAI o1 and Claude with extended thinking enabled.
Markdown file (.md)
A plain-text file format that uses simple symbols to indicate formatting — e.g., **bold**, # Heading. Widely used for documentation and README files on GitHub. The CLAUDE.md file in this project is a Markdown file.
MCP (Model Context Protocol)
A standard that lets AI tools like Claude connect to external services — databases, apps, files, APIs — during a conversation. Think of it as a plugin system that extends what AI can access and do without you manually copying and pasting information. Used in advanced agentic AI workflows.
Model
The specific version of AI being used (e.g., Claude Sonnet, GPT-4o). Different models have different strengths, speeds, and context window sizes. Free tiers typically provide capable but not the most current models.
MoE (Mixture of Experts)
An AI architecture where different specialized sub-networks ("experts") handle different types of inputs, with a routing system selecting which experts to use for each query. Allows larger, more capable models without proportionally increasing computing costs. Used in GPT-4 and some versions of Gemini.
Prompt
The instruction or question you type to an AI. The quality of the output depends heavily on the quality of your prompt — which is why this course teaches prompt templates.
RAG (Retrieval-Augmented Generation)
A technique that improves AI responses by first searching a database of documents for relevant information, then using that context to generate an answer. Used in AI systems that need access to up-to-date or domain-specific knowledge beyond what the model was trained on — for example, an AI that can search your institution's policies before responding.
Single-file HTML
An HTML document that contains all its own CSS and JavaScript, requiring no external files, internet connection, or server to run. Open the file — it works. This is the format all tools in this course use.
Skill
In Claude Code, a pre-built reusable prompt or action that can be triggered with a slash command (e.g., /commit). Skills extend what Claude Code can do beyond its defaults. Not relevant to the browser-based AI workflow used in this course, but useful if you move to Claude Code later.
State machine
A programming pattern where a system moves through defined steps in a fixed order. The AED simulator uses one: Power On → Attach Pads → Analyze → Shock → CPR. Well-suited to simulation tools that follow a clinical protocol.
Temperature
An AI setting controlling how creative vs. precise the responses are. Low temperature = consistent, predictable output. High = more varied and creative. Most agentic AI tools set a reasonable default — you don't need to adjust this.
Terminal
A text-based interface for sending commands directly to your computer's operating system. Instead of clicking icons, you type instructions. Required for running local servers, using Claude Code, or managing files at scale. Not needed for this workshop's single-file HTML approach.
Token
The basic unit AI models process — roughly ¾ of a word. Context windows are measured in tokens (e.g., 200,000 tokens ≈ ~150,000 words). Pasting a large HTML file into a chat consumes many tokens.
Vector database
A type of database that stores data as numerical vectors — lists of numbers representing meaning — to enable fast similarity searches. Used in RAG systems to find relevant documents quickly. Not needed for the tools built in this course, but relevant in advanced AI application development.
Vercel
A free hosting platform that automatically deploys web files from a GitHub repository. Connect your repo, click Deploy, and Vercel gives your tool a live public URL — usually within 30 seconds. Used in Lesson 7. Updates automatically whenever you push a new file to GitHub.
Vibe coding
Describing what you want in natural language and letting AI write the code, without writing code yourself. You are the domain expert; the AI handles the syntax. The entire workflow in this course is vibe coding.
Web Speech API
A browser-native feature that converts text to spoken audio. Used in the AED simulator for voice prompts — no external library needed, no internet required after the file is opened.

Lesson 1

What Is Just in Time Fidelity?

Your job is to know simulation. The AI's job is to know code.

~10 min
Facilitator Tips — Lesson 1

Timing: ~10 minutes for a group walkthrough, or 5 minutes solo.

Discussion prompt: "What's a scenario where a quick custom tool would have saved you time?" Capture 2–3 ideas on a whiteboard — you'll reference them during the build activity.

Framing: "You've been building just-in-time solutions your whole career — a quick moulage, a printed checklist, a prop from the supply closet. This is the same instinct, supercharged."

Key point: Learners don't need to understand code. They need to understand simulation. That's why they're in the room.

Learning Objectives

  • Define "Just in Time Fidelity" in your own words
  • List 3 things an HTML simulation tool can do
  • Understand why single-file HTML resources work well for AI vibe-coding

The Core Idea

In this workshop, you'll use Claude, ChatGPT, or Gemini as a coding partner. You describe what you want — it writes the code. No coding knowledge required.

High-fidelity simulation tools — mannequins, moulage, task trainers — are expensive and require lead time. Just in Time Fidelity is the practice of creating purpose-built, scenario-specific tools quickly and cheaply, exactly when they're needed.

Agentic AI tools can now generate working HTML web apps in seconds. A browser is all learners need to run them. This unlocks a new category of simulation support: lightweight, interactive, instantly deployable digital tools.

This isn't a replacement for high-fidelity simulation — it's a complement. A quick custom tool built for your scenario, for your learners, in the time it takes to write a paragraph.

Why This Matters for Simulation Education

  • Time: A tool that would take a developer days to build can be generated in hours by a non-technical educator.
  • Cost: Free AI tools + a browser = zero marginal cost per tool.
  • Specificity: Purpose-built for your scenario, your learners, your objectives.
  • Iteration: Didn't get it right? Ask AI to change it. Get the full updated file. Test again.
⚠️
Caution — Check your institution's IT policies before deploying AI-generated tools on clinical networks, shared devices, or in patient-facing environments. Most institutions have policies around custom software and web-based tools — a quick conversation with your IT or education team before the workshop can save headaches later.
⚠️
You are responsible for clinical accuracy. AI will generate plausible-sounding content confidently — including drug doses, procedural steps, and clinical values. Always verify that the content in your tool reflects current guidelines and is appropriate for your learners before using it for training.
⚠️
Do not assume these tools are secure. AI-generated code is not vetted for security. The tools built in this workshop do not collect or transmit learner data on their own — but running them on institutional or enterprise hardware, or publishing them to the web, still requires review by your institution's IS or IT department. A publicly accessible URL is convenient, but "accessible" is not the same as "approved." Check before you deploy in a clinical or institutional setting.

What HTML Is — In Plain Language

HTML (HyperText Markup Language) is the language all websites are built with. An HTML file is just a text file that your browser knows how to display. No server, no app store, no installation. You open the file — it works.

Because everything lives in a single file with no external dependencies, agentic AI can generate the entire tool in one response and hand it to you ready to use. That's what makes this approach so fast.

You will not be writing HTML in this course. You will be describing what you want to an AI, and the AI will write it for you. Your job is to know simulation. The AI's job is to know code.

What HTML Simulation Tools Can Do

  • Step-by-step procedural coaching (like the AED simulator in Lesson 2)
  • Interactive checklists and decision trees
  • Timer-based scenarios with alerts
  • Patient assessment screens with vital signs
  • Medication dosage calculators
  • Visual anatomy or equipment diagrams with clickable labels
  • Team role assignment wheels and Crisis Resource Management (CRM) role cards

Why HTML?

Other formats seem like obvious options — but each has a catch:

  • Native apps (iOS/Android) — require App Store approval, developer accounts, and ongoing updates
  • Web frameworks (React, etc.) — need a server, build tools, and ongoing maintenance
  • PowerPoint — no real-time timers, no voice prompts, no dynamic branching

HTML has a different profile:

  • Opens in any browser — no installation required
  • AI generates the entire file in one response
  • Works offline after the first open
  • Free to share — email it, post it, put it on a USB drive
  • Instantly modifiable — paste back into AI, ask for changes, save again

HTML is the lowest-friction path from "I have an idea" to "this is running on someone's screen."

Why Prompts Matter

AI is powerful but literal. The clearer your instructions, the better the output. You don't need to know how to code — but you do need to describe what you want clearly. These 5 templates cover almost every situation you'll encounter.

1
Start From Scratch Use when: starting a new tool
Create a single-file HTML simulation tool for medical training. What it does: [describe the scenario in plain language] Who will use it: [learner type, e.g. nursing students, paramedics] Key features I need: - [feature 1] - [feature 2] - [feature 3] Keep it simple and easy to use. Tip: More detail = better results. Describe the scenario like you're briefing a colleague who has never seen it before.
2
Make It Better Use when: the output is close but not quite right
This is good, but I'd like to make a few changes: - [specific thing to change] - [specific thing to add] - [specific thing to remove] Please update the code and give me the full file again.
Tip: Always ask for "the full file again" — this gives you a complete, working file to copy/paste and save.
3
Add a Feature Use when: you want one specific new capability
Add [specific feature] to this HTML file. [Paste your current HTML code here] The feature should: [describe exactly what it should do]. Please give me the complete updated file.
Tip: One feature at a time works better than listing many at once. You can always add more after.
4
Fix the Bug Use when: something isn't working
Something isn't working in my HTML file. What I expected to happen: [describe it] What actually happens: [describe it] [Paste your current HTML code here] Please find and fix the problem, then give me the full corrected file.
Tip: Describing the gap between "expected" and "actual" is the single most useful thing you can tell a debugging AI.
5
Explain This Code Use when: you're curious or want to learn
Explain this HTML file to me in plain language. I have no coding background. Focus on: [the part you're curious about, e.g. "how the timer works"] [Paste your HTML code here]
Tip: Understanding even a little of what the code does helps you write better prompts next time.

Understanding the Context Window

Every AI conversation has a memory limit. The context window is how much the AI can "hold in mind" at once — your full conversation history, the code you've pasted, and its own responses all count toward this limit.

As the context window fills, AI performance degrades. Think of it like a whiteboard: the more old notes covering it, the harder it is to focus on the current problem. Around 40% capacity, you may start to notice slower or less accurate responses.

The Quadratic Problem (Plain Language)

It's not just that more conversation = more to read. Each new message has to "connect" with every previous message, so the work the AI does grows much faster than the conversation length itself.

A Clinical Analogy

Think of it like a physician joining a patient handoff mid-shift. If they're carrying mental load from ten previous patients, their focus on the current case suffers. A clean handoff — fresh context — leads to sharper attention.

One Session Per Task

Start a fresh AI conversation for each new tool you're building. Don't chain a chest tube checklist session into a medication calculator session in the same chat.

When to start fresh: If responses feel "off," the AI seems confused about what you're building, or you're starting a completely different scenario — open a new chat.

Using the Developer Console

A built-in browser tool that tells you exactly what's broken — and gives you what you need to ask AI to fix it.

Facilitator Tips — Lesson 8

Framing matters: Introduce the console as a "diagnostic instrument" — like a check engine light. Emphasize that learners don't need to understand the code on screen, only find and copy the error message. This removes the intimidation of looking at code.

Demo first: Before sending learners to try it themselves, open aed-app-broken.html on the projector, show the blank page, open the console live, and walk through finding Error 1 together. The visual of opening the console for the first time is high-value.

Learning Objectives

  • Open the developer console in Chrome and Safari
  • Read an error message and identify its type, message, and location

What Is the Developer Console?

Every modern web browser has a developer console built in — free, no install needed, always available. Most people never know it's there.

When an AI-generated tool has a problem, the console is usually the first and best place to look. It does three things:

  • Shows errors — code that couldn't run, displayed in red
  • Shows warnings — code that ran but might cause issues, in yellow
  • Accepts commands — you can type directly and the page responds instantly

Think of it like the check engine light in a car. It doesn't fix the problem — but it tells you something specific is wrong and points you toward where to look.

Browser Shortcut

Safari instructions (extra steps required)

Safari hides its developer tools by default. You need to enable them first:

  1. Open Safari menu → Settings (or Preferences on older Macs)
  2. Click the Advanced tab
  3. Check "Show features for web developers"
  4. Now open the console: Cmd + Option + C
    Or use the menu: Develop → Show JavaScript Console

Reading What You See

Color tells you severity:

  • 🔴 Red — an error occurred; something couldn't run
  • 🟡 Yellow — a warning; something ran but might cause problems
  • ⚪ White/Gray — informational log; usually safe to ignore

Anatomy of an error — three parts to read:

Type
SyntaxError
Message
Unexpected token '{'
Location
aed-app.html:142

The location (filename:line) is a clickable link — clicking it jumps directly to that line of code in the browser's source viewer. You don't need to understand the code to find where the error is.

Three Types of Errors to Know

You don't need to memorize these — just recognize the pattern when you see it.

SyntaxError

A typo or structural problem. The code couldn't even be read. The app usually won't load at all — you'll see a blank or frozen page.

ReferenceError

Something was called that doesn't exist — a function name that was misspelled or deleted. The app loads, but a button or action does nothing.

Logic Error (no red text)

The code runs fine, but the app behaves incorrectly — a timer counts down too fast, a metronome plays at the wrong speed, a value is off. No error appears in the console. You have to notice it by using the app and then investigate by typing commands in the console.

The 4-Step Framework

Every time something breaks, use this sequence:

  1. Find it Open the console. Look for red text. If the page is blank, the console is the only way to know why.
  2. Read it Note the error type, the message, and the file:line location. These three pieces are everything you need.
  3. Locate it Click the filename:line link in the console. It jumps to that exact line of code. Copy a few lines around it — you don't need to understand it.
  4. Fix it with AI Paste the full error message and the surrounding code into your AI tool. Say: "I got this error in my HTML tool. Here's the error and the code near it. Can you find the problem and give me the corrected code?"

Troubleshooting Guide

Reference resource for when things go wrong. Jump here anytime.

Facilitator Tips — Lesson 9

Most common issues in live workshops (in order):

  1. File saved as .html.txt — most common on Mac TextEdit and Windows Notepad
    Mac: Finder → Settings → Advanced → check "Show all filename extensions" so learners can see what's actually saved.
    Windows: File Explorer → View → Show → File name extensions.
  2. AI gave partial code, not a full file — fix with the DOCTYPE prompt above
  3. Voice not working in Safari — Safari is less reliable for tools that use voice or audio. If a learner is on Safari, ask them to switch to Chrome or Edge before troubleshooting further.

How to triage quickly: Ask "Can you open the file in Chrome?" If yes, the file is fine and the issue is browser-specific. If no, the save step went wrong.

Group troubleshoot: Walk through the bug description approach projected on screen — it's valuable for everyone to watch the debugging process live.

Learning Objectives

  • Diagnose the most common HTML generation problems
  • Apply the correct fix without starting over
  • Know when to ask AI for help vs. try a different approach

How to Use This Guide

Find your symptom below. If none match, describe what you expected vs. what you see, paste your code, and ask the AI to find the problem.

❌ The file won't open / browser shows an error
Make sure you saved with a .html extension — not .html.txt. Mac TextEdit: Format → Make Plain Text before saving. Windows Notepad: Set "Save as type" to All Files and type myfile.html — or use quotes: "myfile.html".
❌ The AI gave me only a code block, not a full file
Ask: "Can you give me the complete, self-contained HTML file starting with <!DOCTYPE html> so I can save it directly?" The AI will provide the full file.
❌ Voice prompts aren't working
Check that your browser isn't blocking audio (look for a 🔇 icon in the address bar). Try Chrome or Edge — they have the most complete Web Speech API support. Safari works on Mac but may need a user gesture first (click anything on the page).
❌ The tool looks broken or the layout is wrong
Describe exactly what looks wrong (e.g., "buttons are overlapping," "text is too small on mobile"). Paste your full code and ask: "Fix the layout issue where [describe problem]. Give me the complete updated file."
❌ I edited the code manually and now nothing works
Paste your broken code and ask: "I edited this HTML manually and it broke. Please fix it and return the complete working file." If it's beyond repair, ask: "Rewrite this so it does [describe what it should do]."
❌ The AI stopped halfway through the code
Type "Continue" or "Please complete the rest of the file." Copy both halves and combine them into one file before saving.

General Troubleshooting Mindset

  1. Pause before you type. Most problems come from unclear communication, not from the AI being broken.
  2. Describe the gap. What did you expect? What did you actually get? That gap is the bug.
  3. Paste the code. Never describe a code problem without including the code. The AI can't help without it.
  4. One fix at a time. Ask for one specific change. Stacking multiple requests in one message leads to partial fixes.
  5. If stuck, reset the conversation. Open a new chat, paste your code fresh, and describe the single thing you want fixed.

Why a URL Instead of a File?

  • A permanent link that never changes — bookmark it, share it, put it in your LMS
  • No download needed — learners open it in any browser on any device
  • Auto-updates when you push changes — no need to re-share the file
  • One link for your whole department, cohort, or conference
Once deployed, your tool has a URL like your-project.vercel.app. Share it anywhere — email, QR code, LMS, slide deck.

What Is GitHub?

GitHub is online storage for files — with a complete history of every version you've saved. Think of it as Google Drive for code files, but with the ability to track and revert every change.

You don't need to understand "git" or use the terminal. You can upload files directly through the website.

Create a free account at github.com.

Step 1 — Create a GitHub Repository

A repository is a folder on GitHub that holds your files.

  1. Go to github.com → sign up or sign in (free)
  2. Click the + icon (top right) → New repository
  3. Give it a name (e.g., sim-tools)
  4. Set visibility to Public — Vercel needs read access
  5. Check "Add a README file"
  6. Click Create repository

Step 2 — Upload Your HTML File

  1. Inside your repository, click Add file → Upload files
  2. Drag your .html file into the upload area
  3. Rename it index.html if you want it at the root URL (e.g., yourproject.vercel.app instead of yourproject.vercel.app/my-tool.html)
  4. Add a commit message: "Upload simulation tool"
  5. Click Commit changes
Make sure your file is saved as .html before uploading. The same rules from Lesson 4 apply.

What Is Vercel?

Vercel is a free hosting service that watches your GitHub repository and automatically deploys changes to a live URL.

When you update a file on GitHub, Vercel updates your live URL automatically — usually within 30 seconds. No manual re-deploy needed.

Create a free account at vercel.com — sign in with your GitHub account.

Step 3 — Deploy with Vercel

  1. Go to vercel.com → sign up with your GitHub account
  2. Click Add New Project
  3. Find your repository in the list → click Import
  4. Leave all settings as default → click Deploy
  5. Wait ~30 seconds → Vercel gives you a URL (e.g., sim-tools-abc123.vercel.app)
  6. Click the URL — your tool is live

Updating Your Tool

When you want to improve your tool:

  1. Make changes in your AI tool (Claude, ChatGPT, or Gemini)
  2. Save the updated file
  3. Go to your GitHub repository
  4. Click Add file → Upload files
  5. Upload the updated file with the same filename
  6. Click Commit changes — Vercel auto-deploys within 30 seconds
The URL never changes. Anyone who bookmarked it will automatically see the updated version.
